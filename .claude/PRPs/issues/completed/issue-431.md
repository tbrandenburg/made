## Problem Statement

The Python backend (`packages/pybackend`) has two security vulnerabilities in its dependency tree: **starlette 1.0.0** (CVE-2026-48710 — HIGH, authentication bypass via unvalidated Host header) and **idna 3.11** (CVE-2026-45409 — MEDIUM, denial of service via crafted inputs). Both were detected by `pip-audit` during automated security scanning. The fix requires upgrading starlette to >=1.0.1 and idna to >=3.15.

---

## Analysis

### Change Rationale

Two CVEs were detected in the locked Python dependencies. Both are in packages that are either direct or transitive dependencies of the FastAPI backend. For each, a patched version is available that resolves the vulnerability without API changes. The standard uv-based workflow (`uv lock`) will resolve and pin the safe versions.

- **starlette**: Transitive dependency via `fastapi 0.135.2` -> `starlette`. Not listed in `pyproject.toml` directly. `uv lock` will pull starlette>=1.0.1 automatically when allowed.
- **idna**: Direct dependency in `pyproject.toml` with constraint `>=3.7`. Minimum must be bumped to `>=3.15` to resolve CVE-2026-45409.

### Evidence Chain

WHY: `pip-audit` reports 2 known vulnerabilities in locked dependencies
↓ BECAUSE: starlette 1.0.0 has CVE-2026-48710 (BadHost — allows Host header injection to bypass path-based auth)
Evidence: `uv.lock:1085-1095` — starlette locked at version 1.0.0

↓ BECAUSE: idna 3.11 has CVE-2026-45409 (DoS via crafted Unicode input to `idna.encode()`)
Evidence: `uv.lock:540-546` — idna locked at version 3.11

↓ ROOT CAUSE: Dependency version constraints in `pyproject.toml` and `uv.lock` pin to vulnerable versions
Evidence: `pyproject.toml:22` — `idna>=3.7` (allows 3.11); starlette not in pyproject.toml (resolved to 1.0.0 by uv)

### Integration Points

- `uv.lock` is consumed by `uv sync` (called by `make install-pybackend`, `make test`, `make unit-test`, `make security-audit`, and all CI workflows)
- `fastapi` (pyproject.toml:8) has a transitive dependency on `starlette`; updating `uv.lock` automatically resolves the new starlette version without changes to `pyproject.toml` for starlette
- `idna` is also a transitive dependency of `requests` (uv.lock:1041), so both the direct constraint and the transitive resolution must be satisfied
